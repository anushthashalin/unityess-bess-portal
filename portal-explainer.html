Why this portal exists

Ornate Solar sells UnityESS BESS units (125 kW / 261 kWh each) to C&I clients across India. Before this portal, the team tracked deals in spreadsheets, proposals were manually typed in Word, and follow-ups were forgotten. The portal replaces those workflows with a single React SPA backed by a Node/Express API and a Neon PostgreSQL database, deployed entirely on Vercel.

Opportunity
A potential BESS sale at a specific company. Lives in bess.bd_opportunities, has a stage (Kanban column), a capacity in kWh, and an assigned BD exec.
Follow-up
A scheduled action (call, meeting, site visit) tied to an opportunity. Auto-created when logging an activity with a next_action_date.
Proposal
A commercial quote generated from the Configurator. Contains CAPEX, IRR, payback. Linked to both a client and a project via FK in both directions.
Project
A confirmed or scoping-stage BESS installation. Has a client_id and optionally a proposal_id. Separate from the BD pipeline — a project means execution is likely.
BD Pipeline
The 6-stage Kanban board (First Connect → PO Received). Moving a card calls PATCH /api/bd/opportunities/:id with a new stage value.
Product Type
bess or epc. Used to filter API responses. The portal has two nav trees (BESS BD, EPC BD) served by the same backend, filtered by this field.

    Easy misread to prevent:
    The BD Pipeline and the Projects page are separate workflows. An opportunity in "PO Received" stage does NOT automatically create a project — a project is created separately (often in parallel) and manually linked to a proposal. State & Invariants

  
What must always be true

  
These are the architectural constraints the portal relies on — breaking them causes data integrity issues or silent failures. source inferred


  

    

      
Follow-ups always belong to an opportunity source

      
bd_follow_ups.opp_id is a non-null FK to bd_opportunities.id. A follow-up with no parent opportunity cannot exist. The AddFollowUpModal enforces this with a required opp selector.

    

    

      
Client filter is implicit via product_type source

      
Every API call from the BD pages passes ?product_type=bess (or epc). The same bd_opportunities table serves both product lines — the filter is the only separation.

    

    

      
Proposal linkage is bidirectional but not enforced inferred

      
proposals.project_id and projects.proposal_id both exist, but there is no DB trigger keeping them in sync. PATCH to /api/bess/projects/:id sets proposal_id; the reverse is not auto-set.

    

    

      
BD CRUD routes have no authentication source

      
All /api/bd/* read and write routes are open — no JWT required. This is intentional for the internal tool but means any request can mutate BD data. Only /api/bd/import/* and /api/bd/automation/run are auth-gated.

    

    

      
Map bubbles degrade gracefully source

      
If requirement_kwh is null or 0 for all clients in a state, the Leaflet bubble shows client count instead of capacity. Radius calculation: totalKwh > 0 ? Math.min(12 + Math.floor(totalKwh/100), 36) : Math.min(12 + sc.length*4, 28).

    

    

      
Today's Schedule is client-side filtered source

      
The Dashboard widget calls bdApi.followUps() with no date params — fetches all pending follow-ups — then filters client-side to due_date <= today. There is no server-side "today" endpoint.
